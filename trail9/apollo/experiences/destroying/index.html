Destroying Data

Make the user feel safer

People make mistakes. Sometimes they delete the wrong data. The best we can do is reduce the likelihood they are making a critical, irrevocable, mistake.

When designing an interface that deletes or destroys data, there are several principles to consider:

  1. We want the user to feel confident they won't accidentally destroy data that should not be destroyed.
  2. We want the user to feel confident they have actually destroyed data when that is their intention.
  3. We want to minimize annoyance when the user needs to delete a lot of data.

Relative Ease of Deletion

How should we indicate whether a record can be deleted? In general, the friction required for a user to destroy data should be commensurate with the consequences of accidentally deleting the data. Make it easy to destroy data that is transient or easy to replace, or when the primary goal of an interface is deletion. Require confirmation, conversely, if recreating the data is difficult or time-consuming, or if accidental deletion has serious consequences.

Reducing Fear

There is a psychological aspect to designing a data-deletion scenario. To reduce a user’s fear of accidentally deleting the wrong data, consider the following questions about your user experience:

  • Is there an opportunity to confirm or cancel the deletion?
  • Is there an option to undo deletion?
  • Does the data really need to be deleted? Would an archive option be better?

Your design can also mitigate a user’s fear that data wasn't actually deleted:

  • What messages are there to confirm the successful deletion of a record?
  • How do we handle server-side inability to delete the record (for example, a system failure)?
  • • How do we handle server-side rejection of the deletion request for example, an authorization failure)?

Reducing Annoyance

Unless the data is ephemeral or trivial to recreate, deletion should never be a single-click process: it should always require confirmation. Increasing the friction required to delete something reduces worry. The user doesn’t fear they will accidentally delete something.

One way to design this type of interaction is to hide deletion from the default state of an interface, and require the user to take an action that makes deletion possible. This switches the application to a state where the specific goal of the interface is deletion, and the user must also confirm the deletion. Such an interaction draws a nice balance between friction and user confidence.

However, you also want to reduce the user’s frustration and blind clicking when they need to perform multiple deletions (instead of making them click DELETE over and over again). To that end, wherever possible, make it easy to delete multiple items at once.

For example, in a list view of Widgets, provide an action to “Edit” the list. When the user is in list editing mode, make it possible to select multiple Widgets and delete them together.

For extra confirmation of particularly destructive actions — such as destroying significant data that cannot be restored — you can use a focus-stealing modal confirmation. Note that this is not appropriate in all cases, only when it is appropriate to interrupt the user's flow of action.
